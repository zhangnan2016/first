# TDD 测试驱动开发要求

模型决策：当用户请求实现功能代码、修复 Bug 或重构时应用。

## 核心循环：RED-GREEN-REFACTOR

所有功能代码实现必须遵循 TDD 三步循环，严禁跳过任一阶段：

### 1. RED — 先写失败的测试

- 在编写任何实现代码之前，先编写描述目标行为的测试
- 运行测试，确认它因正确的原因失败（而非编译错误或环境问题）
- 测试名称应清晰表达意图：`should_return_X_when_Y`

### 2. GREEN — 写最少代码让测试通过

- 编写刚好能让测试通过的实现代码，不多不少
- 禁止在此阶段添加测试未覆盖的功能或「以防万一」的分支
- 目标是快速变绿，而非写出完美代码

### 3. REFACTOR — 重构优化

- 测试通过后，在保持绿色（测试通过）的前提下重构代码
- 消除重复（DRY）、改善命名、提取函数、简化逻辑
- 每次重构后立即运行测试，确保行为不变

## 循环规则

```
RED → GREEN → REFACTOR → RED → ...（直到功能完整）
```

- 每个循环应非常短（分钟级），不要一次性写大量代码
- 一次只处理一个行为或一个用例

## 测试覆盖要求

每个功能必须覆盖以下三类场景：

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| **正常路径** | 输入合法时预期行为 | 分页查询返回正确数据 |
| **边界值** | 边界与极限情况 | 空列表、第 0 页、最大值、null |
| **错误路径** | 非法输入或异常状态 | 参数缺失、权限不足、资源不存在 |

## 适用范围

以下场景**必须**遵循 TDD：

- 新功能开发
- Bug 修复（先写复现测试，再修复）
- 重构（测试是重构的安全网）

以下场景**可以**豁免：

- 纯配置文件变更（如 `application.yml`、`.env`）
- 样式与格式调整（不影响逻辑）
- 探索性原型或 spike（验证后丢弃）

## 禁止事项

- **禁止** 先写实现再补测试（Test-After 不是 TDD）
- **禁止** 编写永远不会失败的测试（无断言或断言恒真）
- **禁止** 为了通过测试而修改测试期望值（除非需求确实变更）
- **禁止** 在一个测试中断言过多无关行为（单一职责）
- **禁止** 忽略失败的测试继续写新功能
