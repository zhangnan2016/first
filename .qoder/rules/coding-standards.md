# 编码规范

始终生效，适用于所有智能会话和行间会话请求。

## 通用原则

1. **先测试，后代码** — 遵循 RED-GREEN-REFACTOR 循环，写代码前先写失败的测试
2. **YAGNI** — 你不会需要它，不要编写当前用不到的功能
3. **DRY** — 不要重复自己，提取公共逻辑
4. **单一职责** — 每个函数/类/模块只做一件事
5. **命名清晰** — 变量、函数、类的命名应自解释意图，避免缩写

## 代码风格

- 使用项目已有的代码风格，保持一致性
- 缩进遵循现有约定（不混用 tab 和 space）
- 函数体保持简短，超过 40 行考虑拆分
- 嵌套不超过 3 层，过深时提取子函数
- 文件不超过 300 行，过长时考虑拆分模块

## 错误处理

- 不要吞掉异常，至少记录日志
- 区分可恢复错误与不可恢复错误，分别处理
- 错误信息对用户友好，对开发者有调试价值
- 边界值必须校验（空值、零值、负数、溢出等）

## 注释

- 注释解释"为什么"，而非"做了什么"
- 删除已被代码自身说明的冗余注释
- TODO/FIXME 必须附带 issue 编号或负责人

## 安全

- 永远不硬编码密钥、密码、token
- 用户输入必须经过校验和转义
- SQL 查询使用参数化，禁止字符串拼接
