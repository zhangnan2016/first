# Brainstorming — 苏格拉底式设计提炼

## 何时使用

在编写任何代码之前，当你收到"构建 X"或"加功能 Y"之类的请求时激活。这是**流程技能（优先级最高）**。

## 目标

不要直接跳进写代码。先退一步，通过提问弄清楚用户真正想做什么。提炼出规范后，分段展示供用户阅读和消化。

## 流程

### 1. 理解意图

通过提问澄清：
- 这个功能的**目标**是什么？解决什么问题？
- 谁是**用户/受众**？
- 有哪些**约束**（技术、时间、兼容性）？
- 有没有**已知的替代方案**或参考实现？

**不要假设。** 如果不确定，使用 AskUserQuestion 工具询问。

### 2. 探索替代方案

至少考虑 2-3 种可行方案，简要对比：
- 方案 A：...（优点 / 缺点）
- 方案 B：...（优点 / 缺点）
- 方案 C：...（优点 / 缺点）

推荐其中一种并说明理由，但让用户做最终决定。

### 3. 分段展示设计

不要一次性倾倒一大段设计文档。**分段**展示，每段短到用户能真正阅读和消化：

1. 先展示**范围与目标**，等待确认
2. 再展示**技术方案概要**，等待确认
3. 最后展示**细节与边界情况**

每段后用 AskUserQuestion 确认："这部分看起来对吗？要调整什么？"

### 4. 保存设计文档

设计经用户确认后，保存为设计文档。本项目与 OpenSpec 协同：
- 可运行 `/opsx:propose` 将头脑风暴成果转化为正式变更提案
- 或保存到 `superpowers/specs/` 供 writing-plans 技能使用

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "我大致知道用户想要什么" | 提问确认。假设是 bug 的温床。 |
| "方案很明显，直接做" | 至少提一个替代方案。 |
| "设计文档太长了，一次发完" | 分段。用户读不完长文档。 |
| "用户说'快点做'，跳过设计" | "做什么"不等于"怎么做"。仍需澄清范围。 |

## 与 OpenSpec 协同

brainstorming 的输出可直接喂给 `/opsx:propose`：
- 头脑风暴确定的**意图与范围** → `proposal.md`
- 头脑风暴确定的**技术方案** → `design.md`
- 头脑风暴确定的**需求** → delta `specs/`
