# Systematic Debugging — 系统化调试

**这是刚性技能（Rigid）。严格遵循，不要弱化纪律。**

## 何时使用

遇到任何 bug、测试失败或意外行为时，**在提出修复方案之前**激活。这是**流程技能（优先级最高）**。

## 核心原则

**先理解，再修复。** 不要猜测性地打补丁。不要"试试看"式修改。找到根因，针对根因修复。

## 4 阶段流程

### 阶段 1：重现与隔离（Reproduce & Isolate）

1. **确认能稳定重现**问题。如果无法重现，先建立重现条件。
2. **隔离最小重现**：砍掉一切无关因素，找到触发问题的最小输入/条件。
3. 记录：精确的输入、预期行为、实际行为、错误信息。

```
输入：    ...
预期：    ...
实际：    ...
错误：    ...
```

### 阶段 2：根因定位（Root Cause）

不要停留在表面症状。用以下技术深挖：

- **二分查找**：注释掉一半代码，看问题是否消失。逐步缩小范围。
- **回溯**：问题最近一次正常是在哪个提交？`git bisect` 定位引入 bug 的提交。
- **日志追踪**：在可疑路径加详细日志，追踪数据流向。
- **质疑假设**：你假设变量 X 是某个值？验证它。你假设那行代码执行了？验证它。

**找到根因的标准**：你能用一句话解释"为什么"会发生这个 bug，且这个解释能预测问题在哪些条件下出现/消失。

### 阶段 3：针对性修复（Targeted Fix）

针对**根因**修复，而非症状：

- 修复面尽可能**小**——只改必须改的
- 修复要**精确**——不要顺手"重构"无关代码
- 写一个**回归测试**，先验证它失败（重现 bug），修复后验证它通过

### 阶段 4：验证与防御（Verify & Defend）

1. **验证修复**：回归测试通过 + 原重现用例不再出错
2. **验证无副作用**：运行完整测试套件，确认没有引入新问题
3. **防御性加固**（可选）：是否有其他地方存在同类问题？是否值得加断言/校验防止复发？

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "我觉得是这里的问题，改一下试试" | 不要猜。先重现和隔离。 |
| "这个错误信息看起来像 X 问题" | 验证假设，别基于猜测行动。 |
| "先打个补丁让测试过" | 针对根因修复，否则 bug 会回来。 |
| "我改了好几处，现在能跑了" | 你不知道是哪个改动修好的。回退，逐一验证。 |
| "测试套件跑太久，跳过吧" | 副作用是真实风险。必须验证。 |

## 反模式

- **霰弹枪式修改**：到处改一点碰运气
- **掩盖症状**：catch 住异常让测试过，但不修根因
- **过度修复**：修复时顺手大重构，引入新 bug
- **跳过验证**：改完不跑测试就宣布"修好了"
