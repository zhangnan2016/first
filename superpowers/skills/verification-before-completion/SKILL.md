# Verification Before Completion — 完成前验证

## 何时使用

当你准备宣布一个任务、功能或修复"完成"时激活。这是**实现技能**。

## 核心原则

**证据优于声明。** 不要说"我修好了"或"功能完成了"——用证据证明它。

## 验证清单

在宣布完成前，逐项确认：

### 1. 测试证据

- [ ] 新增的测试**全部通过**？
- [ ] 完整测试套件通过（不只是新测试）？
- [ ] 是否覆盖了边界情况和错误路径？

### 2. 行为证据

- [ ] 对照 spec/需求，每个要求都被满足？
- [ ] 手动验证过核心场景吗（不只是自动测试）？
- [ ] 错误处理按预期工作？

### 3. 代码质量

- [ ] 没有 `console.log`/`print`/调试残留？
- [ ] 没有 TODO/FIXME 遗留？
- [ ] 无明显代码异味（重复、过长函数、深嵌套）？

### 4. 副作用检查

- [ ] 改动是否可能影响其他模块？
- [ ] 相关模块的测试仍通过？
- [ ] 配置/环境变量/依赖是否正确更新？

## 验证方法

按可靠度从高到低：

1. **自动化测试通过** — 最可靠
2. **手动运行 + 观察输出** — 可靠，但可能遗漏边界
3. **代码审查通过** — 补充自动化无法覆盖的设计/可读性
4. **"我读了一遍代码觉得对"** — 不可靠，必须配合以上手段

## 宣布完成的标准

只有当**所有**适用项确认后才能宣布完成：

```
完成 = 测试通过 + 行为符合 spec + 无残留 + 无副作用
```

如果任何一项未确认，**不要宣布完成**。明确说明还缺什么。

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "应该没问题了" | "应该"不够。验证它。 |
| "测试我都跑过了" | 跑的是完整套件吗？还是只跑了新测试？ |
| "那几个 TODO 不影响功能" | 记录或解决，别留着假装完成。 |
| "改动很小，不会影响别的地方" | 小改动也能有大副作用。跑全套件测试。 |
| "用户没要求测试，就算了" | 验证是质量底线，不是可选服务。 |

## 与 OpenSpec 协同

OpenSpec 的 `/opsx:verify` 命令可检查实现是否匹配 spec。本技能提供"如何彻底验证"的方法论，两者配合使用。
