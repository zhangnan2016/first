# Test-Driven Development — 测试驱动开发

**这是刚性技能（Rigid）。严格遵循，不要弱化纪律。**

## 何时使用

在**任何**代码实现期间激活。无论是新功能、bug 修复还是重构。

## 核心循环：RED-GREEN-REFACTOR

```
┌─────────────────────────────────────────────┐
│  1. RED    写一个测试，它应该失败             │
│             运行它 → 看它失败（确认测试有效） │
│  2. GREEN  写最少的代码让测试通过             │
│             运行它 → 看它通过                 │
│  3. REFACTOR 重构代码，保持测试通过           │
│             运行它 → 仍然通过                 │
│  4. COMMIT  提交这个增量                     │
└─────────────────────────────────────────────┘
```

## 详细步骤

### 1. RED — 先写失败的测试

- 根据需求/计划，写**一个**测试用例
- 测试要描述**期望的行为**，而非实现细节
- 运行测试，**确认它失败**，且失败原因是你预期的（不是编译错误或无关原因）

> **为什么必须看它失败？** 如果测试一开始就通过，可能测试本身有 bug（永远断言 true），或者功能已经存在。看它失败证明测试真的在检验某些东西。

### 2. GREEN — 写最少代码

- 写**刚好够用**的代码让测试通过
- 不要写"以后可能用到"的代码（YAGNI）
- 不要追求优雅，先让它过

### 3. REFACTOR — 重构

- 测试已保护你，现在可以放心改善代码质量
- 消除重复（DRY）、改善命名、提取函数
- **每次重构后都运行测试**，确保仍然绿色

### 4. COMMIT — 提交

- 每个 RED-GREEN-REFACTOR 循环后提交一次
- 小步提交，便于回退和审查

## 先于测试写的代码必须删除

如果你发现自己在写生产代码但没有对应的失败测试：**停止。删除那些代码。回头先写测试。**

> 这不是惩罚。先写的代码通常缺少测试覆盖，且往往设计更差（因为没有测试驱动思考接口）。

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "这个太简单了，不需要测试" | 简单的东西最容易写测试。写。 |
| "我先把功能写完，再补测试" | 禁止。先测试，再代码。 |
| "测试失败了但应该是实现的问题" | 先确认失败原因正确，别跳过。 |
| "这个测试很难写，说明不了什么" | 测试难写 = 设计有问题。先改设计。 |
| "重构不用跑测试吧，我没改逻辑" | 每次重构都跑测试。意外无处不在。 |
| "先写个空实现让测试过" | GREEN 是写"最少正确代码"，不是空壳。 |

## 测试反模式

- **快乐路径偏执**：只测正常情况，忽略边界和错误
- **巨石测试**：一个测试断言 20 件事，失败时无法定位
- **实现耦合测试**：测试依赖内部实现细节，一重构就碎
- **永不失败的测试**：断言恒真，提供零价值
