# Using Superpowers

## 核心规则

**在任何响应或动作之前，先调用相关或被请求的技能。** 即使只有 1% 的可能某个技能适用，也要调用它检查。如果调用的技能实际上不适合当前情况，你不必使用它。

## 指令优先级

Superpowers 技能覆盖默认系统提示行为，但 **用户指令始终优先**：

1. **用户的显式指令**（AGENTS.md、直接请求）— 最高优先级
2. **Superpowers 技能** — 在冲突处覆盖默认系统行为
3. **默认系统提示** — 最低优先级

如果 AGENTS.md 说"不要用 TDD"而技能说"总是用 TDD"，遵循用户指令。用户始终拥有控制权。

## 技能优先级

当多个技能可能适用时，按此顺序：

1. **流程技能优先**（brainstorming、debugging）— 决定"如何"着手任务
2. **实现技能其次**（TDD、plans）— 指导执行

```
"让我们构建 X" → 先 brainstorming，再实现技能
"修复这个 bug" → 先 debugging，再领域技能
```

## 红旗警告

这些想法意味着 **停止** — 你在自我合理化：

| 想法 | 现实 |
|---------|---------|
| "这只是个简单问题" | 问题是任务。检查技能。 |
| "我需要先了解更多上下文" | 技能检查在澄清问题之前。 |
| "让我先探索一下代码库" | 技能告诉你如何探索。先检查。 |
| "我能快速检查 git/文件" | 文件缺少对话上下文。检查技能。 |
| "我记得这个技能" | 技能会演进。读取当前版本。 |
| "这不需要正式的技能" | 如果技能存在，就使用它。 |
| "我先做这一件事" | 做任何事之前先检查。 |

## 技能类型

**刚性技能**（TDD、debugging）：严格遵循。不要弱化纪律。

**弹性技能**（模式）：根据上下文调整原则。

技能本身会告诉你它属于哪一类。

## 用户指令

指令说"做什么"，而非"怎么做"。"加 X"或"修 Y"不意味着跳过工作流。
