# Requesting Code Review — 代码审查请求

## 何时使用

在完成任务后、合并/提交前激活。这是**实现技能**。

## 目标

对照**计划和规范**审查代码，发现真实问题（逻辑 bug、安全漏洞、偏离 spec），而非风格之争。

## 审查清单

### 对照计划

- [ ] 实现是否覆盖了计划中的**所有**任务？
- [ ] 是否有**计划外的改动**？（如果有，是否合理？需要更新计划吗？）
- [ ] 文件改动范围是否与预期一致？

### 对照规范

- [ ] 行为是否符合 OpenSpec 的 spec / delta spec？
- [ ] 是否处理了 spec 中的所有场景？
- [ ] 是否引入了 spec 未要求的行为？

### 代码质量

- [ ] 有无逻辑错误或边界遗漏？
- [ ] 有无安全隐患（注入、越权、敏感信息泄露）？
- [ ] 错误处理是否完善？
- [ ] 是否有明显性能问题？

### 测试

- [ ] 新功能是否有对应测试？
- [ ] 测试是否覆盖了关键路径和边界？
- [ ] 测试本身是否有价值（会失败、断言有意义）？

## 问题分级

按严重程度报告问题：

| 级别 | 含义 | 处理 |
|------|------|------|
| **Critical（关键）** | 逻辑 bug、安全漏洞、数据丢失风险、严重偏离 spec | **阻断进度**，必须修复后才能继续 |
| **Major（重要）** | 边界遗漏、错误处理不全、测试缺失 | 应修复，记录为待办 |
| **Minor（次要）** | 命名、注释、轻微重复 | 可选修复，记录建议 |
| **Nit（微调）** | 风格偏好 | 仅作建议，不阻断 |

## 审查输出格式

```
## 代码审查结果

### Critical（必须修复）
- [文件:行] 问题描述 + 建议修复

### Major（建议修复）
- [文件:行] 问题描述 + 建议

### Minor / Nit（参考）
- ...

### 结论
[通过 / 需修复后重新审查 / 阻断]
```

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "代码能跑，应该没问题" | 能跑 ≠ 正确。对照 spec 审查。 |
| "审查太花时间了" | 审查比生产事故便宜得多。 |
| "这是小改动，跳过审查" | 小改动也需要快速审查。 |
| "只关注风格问题" | 风格是 Nit。关注逻辑和安全。 |
| "没有 Critical 就算通过" | Major 问题累积也会出事。 |

## 与 OpenSpec 协同

OpenSpec 的 `/opsx:archive` 前应完成代码审查。归档即表示变更已实现且验证通过，审查是归档前的最后质量关卡。
