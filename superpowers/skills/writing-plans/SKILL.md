# Writing Plans — 编写实现计划

## 何时使用

当有**已批准的设计**时激活（通常来自 brainstorming 或 OpenSpec 的 proposal/design）。这是**流程技能**。

## 目标

把设计变成一份清晰到"一个热情但缺乏判断力、没有项目上下文、讨厌测试的初级工程师"也能跟随实现的计划。

强调：真正的红/绿 TDD、YAGNI（你不会需要它）、DRY（不要重复）。

## 计划质量标准

每个任务必须满足：

### 1. 足够小

- 每个任务 **2-5 分钟**可完成
- 如果一个任务需要超过 5 分钟，**拆分**它
- 宁可拆得太细，不要太粗

### 2. 足够具体

每个任务包含：

- **精确的文件路径**（不是"某个组件"，而是 `src/components/ThemeToggle.tsx`）
- **要做什么**（新建/修改/删除，具体改动）
- **验证步骤**（如何确认这个任务做对了）

### 3. 层级编号

```
## 1. 主题基础设施
- [ ] 1.1 创建 ThemeContext，包含 light/dark 状态
      - 文件：src/contexts/ThemeContext.tsx（新建）
      - 验证：导入不报错，类型检查通过
- [ ] 1.2 添加 CSS 自定义属性
      - 文件：src/styles/globals.css（修改）
      - 验证：浏览器中 :root 变量可访问
```

### 4. 包含验证

每个任务或任务组必须有**验证步骤**：
- 运行什么测试？
- 观察什么行为？
- 检查什么输出？

## 计划结构模板

```markdown
# 计划：<变更名称>

## 前置条件
- [ ] 设计已批准（链接到 design.md）
- [ ] 工作区干净，测试基线通过

## 1. <任务组名称>
- [ ] 1.1 <任务> — 文件：<路径> — 验证：<方法>
- [ ] 1.2 <任务> — 文件：<路径> — 验证：<方法>

## 2. <任务组名称>
- [ ] 2.1 <任务> — ...
- [ ] 2.2 <任务> — ...

## 完成验证
- [ ] 所有任务勾选完成
- [ ] 全部测试套件通过
- [ ] 对照 spec 检查行为符合预期
```

## 与 OpenSpec 协同

OpenSpec 的变更已有 `tasks.md`。writing-plans 技能用于：
- 将 `tasks.md` 进一步细化为可执行级
- 或在 brainstorming 后直接产出实现计划

## 红旗

| 想法 | 纠正 |
|---------|---------|
| "这个任务大概 10 分钟能做完" | 太大了。拆成 2-3 个小任务。 |
| "文件路径到时候再看" | 必须现在写精确路径。 |
| "验证就是'能跑'" | 不够。写明具体测试/观察点。 |
| "先写计划，细节实现时再想" | 计划要包含足够细节，执行时不需要重新设计。 |
